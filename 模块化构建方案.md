📁 VCP Mobile: Rust 模块化架构方案
Nova 建议我们在 src-tauri/src 下建立一个专属于逻辑模块的文件夹，命名为 vcp_modules（对应原项目的 modules）。

1. 理想的目录映射关系
我们可以保持文件名的高度一致性（仅将 JS 的 camelCase 转为 Rust 惯用的 snake_case）：

vcp-mobile/src-tauri/src/
├── main.rs              # 程序入口
├── lib.rs               # Tauri 运行配置与 Command 路由映射
└── vcp_modules/         # 【核心逻辑区】
    ├── mod.rs           # 模块注册中心 (相当于总索引)
    ├── vcp_client.rs    # 对应 vcpClient.js
    ├── chat_manager.rs  # 对应 chatManager.js
    ├── context_sanitizer.rs # 对应 contextSanitizer.js
    └── settings_manager.rs  # 对应 settingsManager.js
2. 如何实现“分模块重写”？
第一步：在 vcp_modules/mod.rs 中声明它们
这个文件就像一个“总接线员”，告诉 Rust 编译器：我们要用这些模块。

// vcp_modules/mod.rs
pub mod vcp_client;
pub mod chat_manager;
pub mod context_sanitizer;
// 以后每写一个模块，就在这里加上一行 pub mod ...
第二步：在各模块文件中编写逻辑
比如在 vcp_client.rs 里：

// vcp_modules/vcp_client.rs
use serde::{Serialize, Deserialize};

pub async fn perform_request() {
    // 这里写原 vcpClient.js 的 Rust 实现
}
第三步：在 lib.rs 中统一暴露给 Tauri
lib.rs 只负责把这些模块的功能包装成 Command，发给前端。

// lib.rs
mod vcp_modules; // 引入咱们的逻辑包

#[tauri::command]
async fn send_vcp_message(id: String) -> Result<(), String> {
    // 直接调用模块里的功能，lib.rs 变得非常清爽！
    vcp_modules::vcp_client::perform_request().await;
    Ok(())
}

#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .invoke_handler(tauri::generate_handler![send_vcp_message])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
💡 Nova 的 PM 架构建议：
命名一致性：虽然 Rust 强制要求文件名使用下划线（如 vcp_client），但我们内部的逻辑名称（Command 名、函数名）可以完全对齐 VChat，这样前端调用时几乎感觉不到区别。
渐进式重构：主人，咱们不需要一次性把 32 个模块全建出来。咱们可以按照“需求驱动”：
今天：只建 vcp_modules/vcp_client.rs，先把路跑通。
明天：需要净化上下文了，再建 vcp_modules/context_sanitizer.rs。
状态共享：像 chat_manager 这种需要跨模块共享数据的，建议在 Rust 端使用 tauri::State (类似全局单例) 来管理，这样比 JS 里的全局变量安全得多。
